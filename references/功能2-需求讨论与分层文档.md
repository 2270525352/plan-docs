# 功能二 · 需求讨论 + 分层文档 SOP

## 目的
站在**产品经理**角度，和用户讨论需求与**产品实用性**（而非一上来就拆开发任务），把讨论结果沉淀为**分层、递归、有索引、互相关联**的文档体系。这是整个技能的主体。

## 核心纪律
- **PM 视角**：主动质疑实用性——"这个功能用户真的会用吗""这里是不是多余了""有没有更简单的达成方式"。先想清楚"为什么做、对谁有用"，再想"怎么做"。
- **细到独立成文**：细到每个功能模块、子模块、甚至"点这个按钮是什么场景、什么逻辑"。哪怕只有几十个字，**也单独成一个文档**。
- **不是碎文件，是网络**：每个文档都要能通过索引和上下游标注，前后追溯、互相关联。

## 分层结构（从宏观到微观，逐层收敛）
1. `00-项目说明书.md` —— 项目是什么、为谁做、解决什么、目标与边界。**源头**。
2. `01-需求文档.md` —— 产品需求总览：功能清单、优先级、用户场景。由说明书推导。
3. `02-架构文档.md` —— 技术架构、技术栈、模块划分、数据流。把需求分配到模块。
4. `03-索引.md` —— 全局导航：模块树 + 横向依赖总表。**单一事实来源**。
5. `modules/` —— 微观实现文档，可无限递归。
6. 扩展：`05-术语表` `06-数据字典` `07-API文档` `08-测试用例`，按项目需要填充。

## 操作步骤
1. **先讨论，后落文档**：围绕用户想法做 PM 式追问，确认目标、用户、场景、实用性。期间持续记原话（功能一）。
2. **自顶向下建骨架**：先写说明书 → 需求 → 架构，明确有哪些一级模块。
3. **建索引**：在 `03-索引.md` 里用缩进画出模块树骨架。**任何结构变更都先改这里。**
4. **逐层细化模块**：对每个模块，在 `modules/模块X/` 下建 `_X.md`（模块说明），再往下拆子模块、拆到叶子（具体交互/按钮）。
5. **标注关联**：每个文档头部写 `上级 / 下级 / 依赖`（模板见下）。横向依赖同时登记到 `03-索引.md` 的依赖总表。
6. **校验通顺**：完成一个模块后，顺着上级→下级链条走一遍，确认逻辑连得上、无断链。

## 关联关系规范（C 方案）
- **纵向**：模块→子模块→子子模块，无限递归，用目录层级体现。
- **横向**：跨模块的调用/触发，用 `依赖:` 字段 + 索引依赖总表双重记录。
- **中心索引**：`03-索引.md` 为准；文档内标注为辅。两者必须一致。

## 文档头部模板
```
上级：[[父模块文档名]]
下级：[[子文档1]]、[[子文档2]]
依赖：[[其它模块-某流程]]   ← 横向，可空
---
（正文：模块用概述+职责；叶子用 场景/触发/逻辑）
```

## 完成标志
用户能从"项目为什么存在"一路点链接追到"这个按钮点了干嘛"，中间不断链。
